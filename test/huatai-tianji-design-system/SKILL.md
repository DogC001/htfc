# 华泰天玑设计系统工作流

## 随包文件

分享此 skill 时，请把整个 `huatai-tianji-design-system/` 目录一起发送，不要只发送 `SKILL.md`。

- `SKILL.md`：任务判断、信息架构、组件映射、执行约束和交付流程。
- `references/huatai-tianji-design-spec.md`：华泰天玑 Web2.0 视觉规范、design tokens、组件状态、布局规则和法务页脚文案。
- `references/share-prompt.md`：给协作者或编码助手的复制即用启动提示词。

使用时必须先读取本文件，再按需读取 `references/huatai-tianji-design-spec.md`。如果只能传一个文件，优先传 `SKILL.md`，但高保真视觉细节会缺失。

## 核心原则

`references/huatai-tianji-design-spec.md` 不是页面生成模板，而是视觉和组件约束。使用规范前，必须先完成页面任务、内容结构、信息层级、组件映射和状态设计。

目标顺序：

1. 先判断页面要解决什么业务问题。
2. 再设计页面放什么、怎么排、谁优先。
3. 然后把内容映射到华泰天玑组件体系。
4. 最后才应用颜色、字体、三层结构、圆角、阴影和交互状态。

## 设计介入级别

开始任务时先判断本次属于哪一级，并在方案中说明：

| 级别 | 适用场景 | 允许行为 |
| --- | --- | --- |
| 保守套用 | 用户要求“保持内容不变”“只美化”“只套规范” | 不新增模块，不改业务文案含义，只优化布局、层级、组件和视觉 |
| 体验优化 | 用户要求“优化页面”“重新设计”“提升体验”“改版”，或原页面存在明显信息缺口 | 可以重组结构、补充常见业务模块、优化文案表达，但必须基于已有内容、业务上下文或用户提供材料合理推导 |
| 产品重构 | 涉及核心流程、信息架构大改、跨页面联动或新增业务能力 | 先输出重构方案和影响范围，等待用户确认后再实现 |

默认规则：

- 若用户说“修改某个页面”“优化页面设计”“重新设计”，优先按“体验优化”处理。
- 若用户明确说“不要改内容”“只调整样式”，按“保守套用”处理。
- 若需要新增业务流程、权限、数据字段、接口或核心功能，升级为“产品重构”，先确认。

## 硬约束

- 未经用户明确允许，禁止引用、复用或改造现成图片、截图、素材图、背景图、插画、图标文件或外部图片资源。
- 未经用户明确允许，禁止为了节省时间而套用、复制、改造现成 HTML、页面文件、模板文件、历史展示页或已有实现文件。
- 可以读取现有文件理解业务、规范和上下文，但不能把它们当作可直接套用的成品。
- 如确实需要参考现成图片、现成页面或历史文件，必须先说明引用对象、用途、影响范围和替代方案，并等待用户明确同意。
- 页面文案只能来自用户提供的真实业务内容、已有业务文档或用户确认的信息；缺失内容不得虚构。
- 不得用“示例数据”“占位文案”“此处替换”“设计说明”等解释型内容填充高保真页面。
- 红、绿、橙只能表达业务语义，不得作为主按钮、菜单选中、tab 选中、分页选中或页面主视觉色。

## 受控主动优化规则

在“体验优化”级别下，允许主动提出或实现以下改进：

1. 信息架构补全
   - 可以基于已有内容补充常见相邻模块。
   - 例如已有“热门课程”和“课程目录”时，可以增加“全部课程”“最近学习”“课程分类”“学习进度”等模块候选。
   - 新增模块必须能从页面目标、用户任务或已有内容中推导出来。

2. 页面结构重组
   - 可以调整内容顺序、分组方式、首屏重点、筛选区、操作区和列表区。
   - 优先让核心任务在首屏可见。

3. 文案优化
   - 可以优化标题、按钮、提示、空状态和说明文字的表达。
   - 不改变业务含义，不新增未经确认的承诺、数据、规则或结论。
   - 文案应更短、更清晰、更适合业务系统使用。

4. 内容补全方式
   - 可新增模块容器、筛选项、tab、空状态和操作入口。
   - 不得编造具体业务数据。
   - 缺少真实数据时，使用真实业务空状态、已有数据源，或标记为“待接入数据”。

5. 明确标注来源
   - 已有内容：来自用户页面、代码或文档。
   - 合理推导：来自页面目标和常见业务流程。
   - 待确认：需要用户确认业务是否存在。

## 输入检查

开始设计前，先确认或从上下文中提取：

- 页面类型：工作台、列表页、详情页、表单页、展示页、弹窗、抽屉、图表页或综合页。
- 设计介入级别：保守套用、体验优化或产品重构。
- 目标用户：业务经理、交易员、运营人员、管理者、客户或其他角色。
- 核心任务：用户进入页面后最需要完成的 1-3 个动作。
- 业务内容：标题、指标、列表、图表、字段、状态、风险提示、操作入口。
- 约束来源：优先读取 `references/huatai-tianji-design-spec.md`；如用户指定其他规范，以用户指定为准。

如果缺少会影响页面结构的关键信息，先提出最少必要的澄清问题。不要用视觉细节掩盖信息架构不清。

## 工作流

### 1. 页面任务定义

先用一句话定义页面：

```text
该页面帮助【用户角色】在【业务场景】中完成【核心任务】。
```

随后列出：

- 主目标：页面必须达成的业务目标。
- 次目标：可以辅助完成但不应抢占主视觉的内容。
- 主操作：页面最重要的操作按钮或入口。
- 风险点：权限、数据缺失、业务风险、误操作或状态异常。

### 2. 内容盘点

把用户提供或文档中的内容拆成结构化清单：

- 页面标题与页面说明。
- 核心指标、关键结论或当前状态。
- 筛选、搜索、时间范围、tab 和快捷入口。
- 表格、列表、卡片、详情字段、图表或流程。
- 操作按钮、文字链接、批量操作、导出、新增、编辑、查看详情。
- 空状态、加载中、错误、权限不足、未登录、未认证。
- 风险提示、法务说明、更新时间、数据来源。

缺失内容要标记为“待用户补充”或设计为空状态，不补虚构业务内容。

### 3. 体验优化机会点

在内容盘点后，必须检查是否存在以下机会：

- 是否缺少总览、全部列表、最近访问、推荐、筛选或分类入口。
- 是否只有局部内容，没有完整任务闭环。
- 是否页面首屏没有主操作或核心状态。
- 是否文案偏说明化、模板化、冗长或不符合业务系统表达。
- 是否信息层级过平，用户无法快速判断重点。
- 是否存在可合并、可拆分或可前置的模块。

若发现机会点，输出：

| 机会点 | 依据 | 建议改法 | 是否需要用户确认 |
| --- | --- | --- | --- |

### 4. 信息层级判定

先给每类内容分配角色，再决定字号、位置和强调方式：

| 内容角色 | 典型内容 | 规范映射 |
| --- | --- | --- |
| 页面一级标题 | 页面名称、业务主题、系统模块名 | H1/H2，页面顶部或主工作区起始位置 |
| 页面说明 | 页面目标、更新时间、数据口径 | 正文或辅助文字 |
| 区块标题 | 概览、明细、趋势、配置、风险提示 | H3/H4 |
| 核心数字 | 关键指标、金额、比例、排名 | H1/H2 或强化数字样式 |
| 字段标签 | 表单项、详情字段、表头 | H6/正文，稳定低噪声 |
| 提示词 | 输入提示、补充说明、弱提示 | H7/辅助文字 |
| 强提示 | 报错、风险、预警、权限限制 | 语义色提示组件 |
| 操作命令 | 查询、重置、新增、导出、提交 | 按钮或文字按钮 |

不要直接凭视觉感觉决定标题大小。必须先确认内容角色，再套用字体层级。

### 5. 低保真布局

先只确定布局，不上高保真样式：

- 选择页面骨架：上下结构、左右结构、主内容加侧栏、看板网格、表格主导或图表主导。
- 判断首屏优先级：首屏必须呈现标题、核心状态、主操作和最重要业务内容。
- 确定承载方式：表格适合高密度明细，卡片适合概览，图表适合趋势和对比，抽屉适合详情或编辑。
- 控制层级数量：不要卡片套卡片；复杂分组优先用分隔线、浅底色、区块标题和栅格。
- 确认响应策略：至少考虑 1920、1440、1366、1280 宽度下的内容收缩方式。

低保真通过后，再进入高保真。不要一开始就套颜色和阴影。

### 6. 内容到组件映射

为每个内容块建立组件映射：

| 内容类型 | 优先组件 |
| --- | --- |
| 页面主标题 | 页面头部标题区 |
| 关键指标 | 指标卡、数字组、趋势摘要 |
| 查询条件 | 输入框、下拉框、日期选择器、筛选 tab |
| 主操作 | 蓝青渐变主按钮 |
| 次操作 | 线框按钮、文字按钮、图标按钮 |
| 状态 | 标签、徽标、状态点 |
| 明细数据 | 表格、分页、批量操作栏 |
| 趋势数据 | 图表、图例、tooltip |
| 详情信息 | 描述列表、抽屉、详情面板 |
| 风险内容 | 风险提示条、warning 状态、说明区 |
| 无数据 | 浅蓝低饱和空状态 |

组件映射完成前，不进入 CSS 或高保真绘制。

### 7. 高保真视觉应用

高保真阶段才应用 `references/huatai-tianji-design-spec.md`：

- 页面底层：浅蓝到浅青背景，允许柔和径向光斑。
- 工作区：半透明浅蓝玻璃层、浅白边框、轻微 blur、柔和蓝灰阴影。
- 内容层：白色或极浅蓝灰 surface，浅描边、轻阴影。
- 主交互：蓝色或蓝青色体系。
- 字体：阿里巴巴普惠体 3.0 优先，正文以 14px/20px 为基础。
- 栅格：1920px 设计基准、1680px 主面板、24 栅格、24px 水槽。
- 圆角：内容卡片常用 8px 或 16px，胶囊控件用 999px。
- 阴影：保持轻，不使用厚重浮层或深色科技舱效果。

只在用户明确允许后，才可以使用已有图片或已有页面作为视觉参考。即使允许，也只能参考，不得无脑套用。

### 8. 状态补全

页面必须覆盖必要状态：

- 按钮：default、hover、active、focus、disabled。
- 输入框：default、hover、focus、filled、error、warning、success、disabled。
- 下拉：closed、open、hover option、selected option、disabled。
- 表格：default row、hover row、selected row、empty。
- tab、菜单、筛选项：default、hover、active、disabled。
- 分页：default、hover、active、disabled。
- 弹窗和抽屉：打开、关闭、加载中、提交中、错误提示。
- 页面级状态：loading、empty、error、no permission、not logged in、not certified。

缺失数据时展示真实业务空状态，不编造数据。

### 9. 设计自检

交付前逐项检查：

1. 页面任务是否一句话说得清？
2. 是否已判断设计介入级别？
3. 页面内容是否来自真实业务材料、用户确认或可说明的合理推导？
4. 主动新增的模块是否没有虚构具体业务数据？
5. 文案优化是否没有改变业务含义？
6. 首屏是否呈现主标题、核心状态、主操作和关键内容？
7. 信息层级是否先判定角色，再映射字号和组件？
8. 每个内容块是否有明确组件归属？
9. 是否遵守浅蓝三层结构？
10. 主交互色是否统一为蓝色或蓝青色？
11. 红、绿、橙是否只用于业务语义？
12. 表格、分页、输入框、按钮、tab、菜单是否有完整状态？
13. 是否没有引用未经允许的现成图片？
14. 是否没有为了省时间套用现成页面、模板或历史文件？
15. 是否没有出现占位文案、示例数据或虚构业务内容？

## 输出格式

如果用户要设计方案，输出：

- 页面任务定义。
- 设计介入级别。
- 内容盘点。
- 体验优化机会点。
- 信息层级表。
- 布局方案。
- 新增/调整模块清单。
- 文案优化清单。
- 内容到组件映射。
- 高保真规范应用说明。
- 状态清单。
- 风险与待确认项。

如果用户要生成页面或高保真代码，先输出简短设计方案并等待用户确认；确认后再实现。

如果用户要求直接修改已有页面，先说明改造范围、是否会触及现成图片或现成文件复用、风险和验证方式。未经确认，不进行图片引用或现成文件套用。

## 分享与移植检查

交付或转发给他人前，检查：

1. `huatai-tianji-design-system/` 目录下是否包含 `SKILL.md` 和 `references/`。
2. `references/huatai-tianji-design-spec.md` 是否可直接打开，且不依赖发送者本地路径。
3. 协作者是否知道先读 `SKILL.md`，再读设计规范 reference。
4. 若目标项目没有阿里巴巴普惠体 3.0、图标库或图表库，需按项目现有字体、图标、图表体系做等价映射，不硬引新依赖。
5. 若需要引用 PDF、截图、历史页面或图片资产，必须另行确认授权和传递范围；本 skill 默认不携带二进制素材。
