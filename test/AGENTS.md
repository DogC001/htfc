# Codex 协作规则

## 总原则

- 必须严格按照用户本次明确指令、UI 规范、参考截图和交付格式执行。
- 本项目主要任务是：依据 UI 规范和现有页面截图，对现有页面进行视觉改版。
- 改版应优先保持原有信息结构、交互逻辑和布局关系。
- 默认只调整样式、颜色、字体、间距、圆角、阴影、图标、视觉层级等表现层内容。
- 如果用户说“其他不要动”，只能修改用户指定区域。
- 如果规范、截图和用户指令冲突，以用户本次指令和截图标注优先；仍不明确时先询问。

## 必读文件

- 开始原型任务前，必须读取 `AGENTS.md`。
- 如任务涉及 HTML 原型改版，必须参考 `DH-skills.md`。
- 如用户指定 UI 规范或设计系统，必须读取用户指定文件。
- 如用户指定参考截图目录，必须查看截图后再生成。

## 原型生成规则

- 生成的原型必须是可编辑 HTML 文件，除非用户指定其他格式。
- 默认优先生成或修改当前项目内文件。
- 新版本优先另存为清晰命名，例如 `xxx-v2.html`、`xxx-v3.html`。
- 不要删除旧版本文件。
- 未经确认，不要覆盖已有文件。
- 页面文案默认使用中文。

## 设计执行规则

- 顶部导航、左侧导航、按钮、表格、表单、弹窗必须优先参考用户给出的最新截图。
- 后台业务页面应保持清晰、克制、高效，避免营销页风格。
- 字号层级必须合理：子页面标题不得大于所属模块主标题。
- 详情页字段行必须使用稳定行结构，避免间距忽大忽小。
- 同类按钮必须统一高度、圆角、颜色和字重。
- 图标不能用过小或难看的文本符号替代，必要时使用 CSS 或 SVG 绘制。
- UI 元素和文字不得重叠、溢出或遮挡。

## 交互规则

- 尽量不改变原有交互方式。
- 左侧折叠菜单如为一级菜单，应优先使用手风琴逻辑：一次最多展开一个。
- 多个菜单项指向同一页面时，只高亮当前点击项。
- 弹窗、Tab、开关、标签选择、勾选启用按钮等交互应可实际点击。

## 交付说明

- 完成后说明创建或修改了哪些文件。
- 说明修改了哪些区域。
- 如果用户要求局部修改，说明哪些区域保持未动。
- 如果运行了检查命令，需要说明结果。
- 如果未能完成某项要求，需要明确说明原因。
