# 华泰天玑设计系统工作流 v2

## 0. v2 定位

`huatai-tianji-design-systemv2` 是在原 `huatai-tianji-design-system` 已经生成并落地过页面的经验基础上扩写的版本。

v2 已经把华泰天玑 Web2.0 UI 规范内容内置整理，采用“两层结构”：

- V2 内置硬规范：颜色、字体、阴影、圆角、按钮、输入、下拉、标签、图标等硬依据。
- 落地经验补充：页面类型、真实代码改样式边界、全系统组件统一和交付检查。

执行时不得用落地经验补充覆盖 V2 内置硬规范。

原版的价值：

- 已验证华泰天玑 Web2.0 的浅蓝、玻璃工作区、白色内容层风格可落地。
- 已验证我的账户等业务页面适合克制、高效、信息密度适中的后台业务视觉。
- 已验证顶部导航、左侧导航、详情字段、表格、表单、弹窗等组件规则方向正确。

v2 的目标：

- 把原版较粗的视觉规范扩展到“全系统页面样式改版”可用。
- 颗粒度参考 `test-skills.md`：页面类型、尺寸、布局、组件、状态、命名、检查清单都要明确。
- 支持两类任务：
  - 生成或改版 HTML / 原型页面。
  - 读取真实天玑前端代码后，只修改前端样式代码。

## 1. 随包文件

分享或迁移时必须发送整个 `huatai-tianji-design-systemv2/` 目录。

- `SKILL.md`：总工作流、权限边界、执行顺序、交付格式。
- `references/huatai-tianji-design-spec.md`：完整视觉规范、tokens、布局、组件、状态。
- `references/page-patterns.md`：全系统页面类型与布局模式。
- `references/component-rules.md`：组件级样式、状态和约束。
- `references/style-code-workflow.md`：真实前端代码中“只改样式”的工作流。
- `references/delivery-checklist.md`：交付检查和改动说明模板。
- `references/share-prompt.md`：可复制给协作者或编码助手的启动提示词。

## 2. 最高优先级原则

1. 用户本次明确指令优先。
2. 用户本次指定的系统代码、目标页面、组件范围、截图、标注和局部修改范围优先。
3. V2 内置硬规范优先于 v2 落地经验补充。
4. 原有业务信息结构、交互逻辑和页面流程优先保持。
5. 真实代码样式改版默认先做“颜色层改版”，不要主动做“布局层改版”。
6. 默认优先修改颜色、背景、描边、横线/分割线、文字颜色、hover/active/disabled 状态、图标颜色、阴影和圆角。
7. 除非用户明确要求或原页面明显破版，不主动修改 `margin`、`padding`、`gap`、`width`、`height`、`line-height`、`display`、`position`、`top/right/bottom/left`、`flex`、`grid`、`align-items`、`justify-content`、`overflow`。
8. 尽量不改原有按钮、弹窗、图片、头像、图标、表格列宽、表单控件、卡片、布局容器的尺寸和对齐关系。
9. 如必须修改宽高、行高、列宽、图片尺寸、弹窗尺寸、按钮高度、布局栅格、边距或对齐，必须先说明原因、范围和风险。
10. 默认只改表现层：颜色、字体、描边、圆角、阴影、图标、视觉层级、状态；间距和布局只在必要时谨慎调整。
11. 真实代码改版时，默认只能修改前端样式相关代码。
12. 不得擅自修改接口、路由、权限、状态管理、业务逻辑、数据结构、表单业务校验、埋点、构建配置和依赖。
13. 若必须新增 class、style hook 或轻微模板标记才能套样式，必须先说明原因、范围和风险，等待用户确认。
14. 完成后必须列举修改了哪些文件、哪些内容、哪些区域未动、验证结果和风险。

## 3. 必读顺序

开始任务前按顺序读取：

1. 当前项目的 `AGENTS.md`。
2. 用户指定的系统代码、页面、组件、截图、规范或修改范围。
3. 本文件 `huatai-tianji-design-systemv2/SKILL.md`。
4. `references/huatai-tianji-design-spec.md`。
5. 根据任务类型读取：
   - 页面整体改版：`references/page-patterns.md`、`references/component-rules.md`。
   - 真实代码改样式：`references/style-code-workflow.md`。
   - 交付前：`references/delivery-checklist.md`。

## 4. 任务类型判断

| 类型 | 场景 | 执行方式 |
| --- | --- | --- |
| HTML 原型改版 | 修改当前项目内 HTML 页面或生成新版本 | 用户未明确要求覆盖时，优先另存新文件 |
| 局部视觉修正 | 用户指定某个区域、组件、代码范围或视觉问题 | 只改指定区域，其它不动 |
| 全页面视觉套规范 | 对已有页面整体按天玑规范改版 | 保持业务结构和交互，重做视觉层 |
| 真实代码样式改版 | 读取天玑前端代码并直接改样式 | 只改样式文件或确认过的最小 style hook |
| 组件体系统一 | 多页面按钮、表格、表单、弹窗等不统一 | 优先改组件样式或主题 token，并说明影响范围 |
| 产品/流程重构 | 涉及业务流程、数据字段、权限或核心交互 | 先输出方案，等待确认，不直接改 |

## 5. 设计介入级别

| 级别 | 适用场景 | 允许行为 |
| --- | --- | --- |
| 保守套用 | “只套规范”“内容不变”“其它不要动” | 不新增模块，不改业务文案含义，只改视觉表现 |
| 体验优化 | “优化页面”“重新设计”“提升体验” | 可调整分组、层级、顺序和状态，但不编造业务内容 |
| 系统化统一 | “全系统页面都要统一” | 建立 tokens、组件规则和页面模式，控制共享影响 |
| 产品重构 | 改流程、字段、权限、路由、接口 | 必须先确认方案 |

默认：如果用户要求“前端样式改版”，按“保守套用”或“系统化统一”处理，不主动改业务结构。

## 6. 页面改版执行流程

### 6.1 先理解页面

必须确认：

- 页面类型：工作台、列表页、详情页、表单页、配置页、图表页、弹窗、抽屉、空状态、权限页等。
- 用户角色：客户、业务经理、交易员、运营、管理员、研究员等。
- 核心任务：进入页面后最重要的 1-3 个动作。
- 当前结构：顶部导航、侧边栏、筛选区、内容区、操作区、表格/表单/详情。
- 数据状态：正常、空、加载、错误、权限不足、未认证、禁用。

### 6.2 再做组件映射

把页面内容映射到天玑组件：

| 内容 | 组件 |
| --- | --- |
| 页面标题 | 页面头部 / 标题栏 |
| 核心指标 | 指标卡 / 数字组 |
| 筛选查询 | 搜索、下拉、日期、Tab、快捷筛选 |
| 明细数据 | 表格、分页、批量操作 |
| 详情信息 | 描述列表、字段行、信息分组 |
| 编辑录入 | 表单、上传、校验提示 |
| 主操作 | 蓝青渐变主按钮 |
| 次操作 | 蓝色描边按钮 / 文本按钮 |
| 风险操作 | 红色语义按钮 |
| 状态 | 标签、徽标、状态点 |
| 弹层 | 弹窗、抽屉、确认框 |
| 无数据 | 浅蓝低饱和空状态 |

### 6.3 最后应用视觉规范

顺序固定：

1. 页面底层：浅蓝到浅青背景。
2. 工作区：玻璃层。
3. 内容层：白色/极浅蓝灰 surface。
4. 字体层级：标题、正文、辅助文字。
5. 组件样式：按钮、表格、表单、菜单、Tab、弹窗。
6. 状态样式：hover、active、focus、disabled、error、warning、success、empty。
7. 响应式和溢出检查。

## 7. 真实代码样式改版边界

真实代码任务中：

允许修改：

- `.css`、`.less`、`.scss`、`.sass`
- `.module.css`、`.module.less`、`.module.scss`
- 主题 token、变量、组件库覆盖样式
- 纯样式对象文件，例如 `styles.ts`、`theme.ts`、`tokens.ts`
- Vue SFC 的 `<style>` 区块

谨慎修改，必须先确认：

- JSX / Vue template 中新增静态 class。
- 仅用于样式定位的 wrapper。
- CSS-in-JS 文件中样式对象。

禁止修改：

- API、路由、权限、状态管理。
- 业务计算、数据处理、表单业务校验。
- 点击事件、弹窗开关、Tab 切换、提交流程。
- 文案语义、mock 数据、依赖、构建配置。

## 8. 输出要求

### 方案阶段

如果用户要求先给方案，输出：

- 页面类型判断。
- 改版范围。
- 涉及组件。
- 采用哪些天玑规范。
- 是否只改样式。
- 风险和待确认点。

### 实现完成后

必须输出：

- 修改文件清单。
- 每个文件修改内容。
- 哪些区域保持未动。
- 是否触碰业务逻辑。
- 检查命令和结果。
- 未完成项或风险。

## 9. 最终自检

交付前逐项检查：

- 是否读取了本 skill 和相关 reference。
- 是否遵守用户本次指令。
- 是否区分了 V2 内置硬规范和 v2 落地经验补充。
- 是否优先使用 V2 内置 token、字体、按钮、输入、下拉、标签、阴影和圆角规则。
- 是否保留原有业务结构和交互。
- 是否只修改样式相关代码。
- 是否默认只做颜色层改版，未主动做布局层改版。
- 是否尽量保留原有按钮、弹窗、图片、表格列宽、表单控件和布局容器尺寸。
- 若改了边距、对齐、宽高、行高、列宽、布局栅格，是否说明必要原因和影响范围。
- 是否检查所有按钮颜色，包括自定义按钮、分享、导出、取消、重置、文本按钮和图标按钮。
- 是否检查普通筛选项、商品 chip、合约选择、分段选择器、Tab、菜单、分页、快捷筛选的选中态，确认未使用红色、粉红或浅红背景。
- 是否进行页面视觉自审，确认没有边距错乱、对齐错乱、元素遮挡或文字溢出。
- 是否符合浅蓝三层结构。
- 主交互是否统一蓝/蓝青体系。
- 红、绿、橙是否只用于业务语义。
- 未读消息徽标、品牌 logo、图表橙色线是否按允许例外处理，未扩散到普通控件选中态。
- 表格、表单、按钮、弹窗、菜单状态是否完整。
- 文本是否溢出、遮挡、重叠。
- 响应式宽度是否可接受。
- 是否列清楚文件和改动内容。
