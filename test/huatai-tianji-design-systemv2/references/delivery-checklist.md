# 华泰天玑 v2 交付检查清单

## 1. 修改前检查

- 是否读取 `SKILL.md`。
- 是否读取 `huatai-tianji-design-spec.md`。
- 是否确认 V2 内置硬规范优先于 v2 落地经验补充。
- 是否判断页面类型。
- 是否确认用户是否要求“只改样式”或“其它不要动”。
- 是否定位目标页面、组件和样式入口。
- 是否判断改动是否影响共享组件。
- 是否确认本次默认采用“颜色层改版，不做布局层改版”。
- 是否列出目标页面所有按钮类型，包括分享、导出、导入、查询、重置、取消、保存、提交、文字按钮、图标按钮和表格操作按钮。

## 2. 视觉检查

- 是否套用了 V2 内置色值。
- 是否正确区分普通主操作蓝色体系和风险语义红色体系。
- 是否检查商品筛选、合约选择、交易所 Tab、快捷筛选、chip、分段选择器的选中态，确认未使用红色、粉红或浅红背景。
- 红色/浅红背景是否只用于风险、失败、报错、上涨、高风险确认等明确业务语义。
- 是否保留了允许例外：未读消息徽标可小面积红色、品牌 logo 可保留品牌红、图表数据序列可保留橙色线。
- 是否在需要新建、明显不统一、原页面破版或用户明确要求时，才补齐按钮 L/M/S 尺寸规则：40px/18px、32px/16px、24px/14px。
- 是否默认保留了原有按钮、弹窗、图片、表格列宽和布局容器尺寸。
- 如修改了宽高、行高、列宽、图片尺寸、弹窗尺寸、按钮高度，是否说明必要原因和影响范围。
- 是否没有无意修改原有边距、对齐、布局栅格、按钮尺寸、弹窗尺寸、图片尺寸。
- 是否检查所有按钮颜色，尤其是分享、导出、导入、重置、取消、保存、查询、文字按钮、图标按钮、自定义按钮。
- “分享”等普通次级操作是否使用蓝色/蓝青体系或中性体系，而不是误用红色风险体系。
- 普通筛选项、商品 chip、合约选择、分段选择器、Tab、菜单、分页、快捷筛选是否使用蓝色/蓝青选中态，而不是红色系选中态。
- 如果是新建、明显不统一、原页面破版或用户明确要求，是否检查输入框、搜索框、日期框、下拉框高度为 40px；否则是否保留原控件高度。
- 下拉框是否符合 V2 内置规则：hover 背景、10 条后滚动、滚动条位置和颜色。
- 标签颜色是否对应业务语义，未把红绿橙用于普通选中。
- 空状态是否区分“暂无数据”和“暂无搜索结果”，且未编造数据。
- 页面是否符合浅蓝三层结构。
- 顶部导航是否稳定、克制。
- 左侧导航是否清晰，选中态是否蓝色体系。
- 主按钮是否蓝青体系。
- 风险、成功、警示是否使用对应语义色。
- 表格行高、表头、hover 是否统一。
- 表单控件高度、边框、focus、disabled 是否统一。
- 弹窗标题、关闭、内容、底部操作是否稳定。
- 详情字段行是否固定结构。
- 空状态是否真实，不编造数据。
- 文本是否溢出、重叠、遮挡。
- 1440/1366/1280 宽度下是否明显破版。
- 是否完成改后视觉自审，检查边距错乱、对齐漂移、元素重叠、文本溢出、弹窗/图片/表格宽度异常。

## 3. 代码改版检查

- 是否只修改样式文件或确认过的 style hook。
- 是否优先修改颜色、背景、描边、横线/分割线、文字颜色和状态，而不是重排尺寸。
- 是否没有主动修改 `margin`、`padding`、`gap`、`width`、`height`、`line-height`、`display`、`position`、`top/right/bottom/left`、`flex`、`grid`、`align-items`、`justify-content`、`overflow`、`transform` 等布局保护属性。
- 如果修改了布局保护属性，是否单独说明原因、范围、风险和视觉验证结果。
- 是否没有修改业务逻辑。
- 是否没有修改接口、路由、权限、状态管理。
- 是否没有新增依赖。
- 是否没有改变交互行为。
- 是否没有改变文案语义。
- 是否控制了共享样式影响范围。
- 是否避免过多 `!important`。

## 4. 检查命令建议

按项目实际情况运行：

```text
npm run lint
npm run typecheck
npm run test
npm run build
npm run dev
```

如果命令不存在或无法运行，说明原因。
